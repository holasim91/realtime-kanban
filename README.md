# realtime-kanban

Supabase Realtime broadcast 기반 실시간 협업 칸반보드.
타임스탬프 기반 LWW(Last-Write-Wins)로 동시 편집 충돌을 처리하고,
낙관적 업데이트 + 롤백으로 반응성을 확보하는 것을 목표로 한 학습 프로젝트.

## 기술 스택

- Vite + React + TypeScript
- 상태: useReducer (정규화된 flat 상태)
- 실시간: Supabase Realtime (broadcast)
- 영속화: Supabase (Postgres)
- 드래그: dnd-kit

## 실행

\```bash
npm install
npm run dev
\```

`.env`에 Supabase 값 필요:

\```
VITE_SUPABASE_URL=your-project-url
VITE_SUPABASE_KEY=your-anon-key
\```

## 설계 노트

### 1. Postgres Changes 대신 broadcast를 선택한 이유

Supabase는 실시간 동기화를 위해 Postgres Changes(DB 변경을 자동 전파)와 broadcast(클라이언트가 직접 이벤트 전송)라는 두 가지 방식을 제공한다.

Postgres Changes를 사용하면 코드는 줄어들지만, 충돌 해소는 사실상 DB 트랜잭션 순서에 맡겨진다. 이 프로젝트의 목표는 동시 편집 충돌을 직접 다뤄보는 것이었다. 따라서 충돌 해소 로직을 클라이언트가 제어할 수 있도록 broadcast를 선택했다.

DB 트리거에서 broadcast를 발행하는 `realtime.broadcast_changes` 방식도 검토했다. 하지만 이 역시 DB가 확정한 최종 상태를 전파하는 구조이므로 같은 이유로 제외했다.

DB는 단일 진실원(source of truth)으로 두되, broadcast는 지연을 줄이기 위한 수단으로 사용한다. 마운트할 때 DB에서 상태를 먼저 불러온 뒤 구독을 시작하는 것도 이 때문이다. broadcast 이벤트가 유실되더라도 새로고침하면 DB 상태를 기준으로 다시 조정된다.

### 2. 낙관적 업데이트의 롤백을 스냅샷 복원이 아닌 역방향 이동으로 구현

처음에는 드래그를 시작할 때 보드 전체를 스냅샷으로 저장하고, 저장에 실패하면 해당 스냅샷으로 되돌리는 방식을 사용했다.

협업 환경에서는 문제가 생긴다. A의 저장이 실패하기까지 수백 ms가 걸리는 동안 B가 다른 카드를 옮기면, A의 롤백이 B의 이동까지 함께 되돌린다. 내 저장 실패 때문에 다른 사용자의 정상적인 변경까지 덮어쓰는 셈이다.

그래서 롤백 방식을 "원래 컬럼·원래 위치로의 역방향 이동"으로 바꿨다. 해당 카드 한 장만 되돌리기 때문에 동시에 발생한 다른 변경은 그대로 보존된다. 두 개의 탭에서 저장 실패를 강제로 재현해 이 동작을 확인했다.

### 3. 카드 위치를 정수 재인덱싱이 아닌 float 중간값으로 관리

카드마다 순서를 나타내는 숫자(position)를 갖는다. 이 숫자를 어떻게 매기느냐에 따라
"한 카드를 옮길 때 몇 개의 카드를 수정해야 하는가"가 달라진다.

**정수 순서(0, 1, 2...)를 쓸 경우.**
0번과 1번 카드 사이에 새 카드를 넣으려면 넣을 자리의 숫자가 없다.
그래서 1번을 2번으로, 2번을 3번으로... 뒤따르는 카드를 전부 한 칸씩 밀어야 한다.
카드 한 장을 옮겼는데 여러 행을 UPDATE하게 되고, 그중 하나라도 실패하면
어디까지 되돌려야 하는지가 복잡해진다.

**중간값을 쓸 경우.**
position을 정수가 아닌 실수(double precision)로 두고, 처음에 1000, 2000, 3000처럼
간격을 띄워 배치한다. 1000번과 2000번 사이에 카드를 넣고 싶으면
두 값의 중간인 1500을 주면 끝이다. 앞뒤 카드는 그대로 1000, 2000이다.
정렬하면 1000 → 1500 → 2000 순서가 나온다.

즉 **옮긴 카드 한 장만 UPDATE하면 된다.** 다른 카드는 손대지 않는다.
그 결과 이동·추가·순서 변경이 전부 "한 행 쓰기"라는 같은 모양이 되고,
실패했을 때 되돌릴 대상도 그 한 행뿐이라 롤백이 단순해진다.

**트레이드오프는 정밀도다.** 같은 자리에 계속 삽입하면
1500 → 1250 → 1125 → ... 간격이 절반씩 줄어든다. 수십 번 반복하면
실수가 표현할 수 있는 한계에 부딪힌다. 이 프로젝트 규모에서는
발생하지 않아 별도의 대응을 하지 않았다.

### 4. 클라이언트 타임스탬프 기반 LWW의 한계

충돌은 Last-Write-Wins으로 해소한다. 각 카드는 `updatedAt`(epoch ms)을 갖고 있으며, 들어온 이벤트의 타임스탬프가 현재 값보다 크지 않으면 오래된 이벤트로 판단해 폐기한다.

이 방식에는 알려진 한계가 두 가지 있다.

**클럭 스큐(Clock Skew)** — 타임스탬프를 클라이언트에서 생성하기 때문에 시계가 앞선 클라이언트가 항상 이긴다. 서버 권위 타임스탬프를 사용하면 해결할 수 있지만, 충돌 해소 권한이 서버로 넘어가므로 1번에서 broadcast를 선택한 이유와 어긋난다.

**텍스트 편집에는 부적합** — 카드 위치는 LWW에 따라 한쪽이 이기고 다른 쪽이 져도 괜찮다. 그러나 같은 텍스트 필드를 동시에 편집하면 진 쪽의 입력이 통째로 사라진다. 이 영역에는 필드 단위 잠금이나 CRDT가 필요하며, 이 프로젝트의 범위를 벗어난다. 현재 카드 제목 편집 기능을 넣지 않은 이유도 여기에 있다.

**삭제와 이동의 충돌** — 삭제에는 타임스탬프 비교를 적용하지 않았다.
한 사용자가 카드를 삭제하는 동안 다른 사용자가 같은 카드를 옮기면,
이동 이벤트가 나중에 발생했더라도 삭제가 우선한다.
(reducer에서 존재하지 않는 카드에 대한 이동 이벤트는 무시하도록 처리했다.)
엄밀히 하려면 삭제 표식(tombstone)을 남겨 삭제 시각과 이동 시각을 비교해야 하지만,
이 프로젝트의 범위를 넘어선다고 판단해 구현하지 않았다.

카드 이동이라는 문제에 한해서는 LWW가 적절하다고 판단했다.
